## Key findings

- Design standards were built primarily **for AI agents, not for designers** — the driving motivation was making harnesses/skills work reliably, with human designer alignment as a secondary benefit (Q4, Q6).
- The standards site is structured as machine-readable input: markdown-first, with a dedicated "for Agents" page and a control catalog indexed to match the underlying markdown files (Q6).
- Quality is prioritized in a specific order: **accessibility > user-friendliness > aesthetics**, but in practice accessibility is not checked systematically — designers trust AI output and their own intuition rather than working through an a11y checklist (Q5). This is a gap between stated standard and actual practice.
- AI-assisted design work is organized around broad, holistic goals (e.g. "improve the design standards site") rather than discrete features, producing large-scope PRs by design (Q7).
- Skill adoption is social/reputational, not systematic — Wondo picked up `get-shit-done` because Ivan and Reza recommended it, not from an evaluation process (Q7).
- Known recurring failure modes: AI doesn't reliably self-select the right skill without explicit prompting ("use relevant skills"); pixel-level visual edits are cumbersome to express in prompts; AI (or the harness) tends to over-engineer when given reference designs (Q7, Q8).
- Context/memory loss across sessions is a known problem — Tasha built a skill to persist design memory (e.g. "don't use XX font, follow the catalog"), but even with it, output still sometimes deviates from standards (Q8).
- Wondo's aspirational skill is a **compounding "loop" skill** — one that improves itself the more designers use it (Q9).

## Steps identified in designer process

1. Set a broad, cross-cutting goal for the work (not a single feature/page) (Q7).
2. Invoke the built-in `/goal` skill to have AI turn that goal into a working prompt (Q7).
3. Explicitly remind the AI to use relevant skills ("please make sure you are using these skills") — self-selection isn't trusted (Q7, Q8).
4. Let AI select and apply the specific skill(s) needed (e.g. `/tfx:copy`, `/tfx:polish`) (Q6, Q7).
5. Use AI to "grill" him with questions to sensemake the problem before/while executing (Q7).
6. Supply reference material (design systems, reference websites) with prompts like "build as good as this" (Q8).
7. Review output for "AI slop" and deviation from the design standards catalog before accepting (Q2, Q8).
8. Rely on senior/AI-savvy designers (Reza, Tasha) to review, refine, and patch gaps (memory, standards) that generalist flows miss (Q2, Q8).

## Candidate skills or harnesses

- `/goal` — turns a broad objective into an actionable AI prompt (Q7).
- `/tfx:copy`, `/tfx:polish` — narrow skills that each improve one specific dimension of a design (Q6).
- [`get-shit-done`](https://github.com/gsd-build/get-shit-done) — general prototyping harness, adopted via peer recommendation; associated with an over-engineering failure mode (Q7, Q8).
- Tasha's memory/context-management skill — persists design decisions/constraints across sessions to reduce drift (Q8). Worth investigating directly — imperfect but closest existing solution to the "gap" problem Wondo describes.
- [tfx-design standards site](https://tfx-design.vercel.app/overview) + ["for Agents" page](https://tfx-design.vercel.app/for-agents) + [control catalog](https://tfx-design.vercel.app/standards/catalog) — not a skill itself, but the machine-readable substrate other skills are meant to read from.
- Proposed/aspirational: a compounding "loop" skill that improves with repeated use (Q9) — not yet built, candidate for future skill-creator work.

## Open questions

- What does Tasha's memory/context skill do mechanically, and why does deviation still occur despite it — is this a prompting gap, a skill limitation, or a model limitation?
- For pixel-level/visual tweaks ("move it 2px left"), is there a tooling gap that a skill could close, or is this inherently better suited to a visual editor than a prompt?
- Does the "for Agents" page / control catalog actually get read and followed by coding agents in practice, or is it aspirational structure that hasn't been validated end-to-end?
- Untested against the original hypothesis: could an engineer, without designer input, actually produce standards-compliant UI using only this harness + catalog — or does it still require a designer's judgment call (e.g. Reza/Tasha) to catch drift?
