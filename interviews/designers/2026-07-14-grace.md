### Process mapping

### Problems
> Grace:

She would often have to take screenshots of the rendered app from the prototype repo and the actual code repo, place them side by side in Figma, and see what the differences are (different font sizes, placements etc.).

## Key findings

1. **Grace's workflow is entirely AI-native, not Figma-native.** She uses Claude Code in a dedicated designer prototype repo to create and iterate on designs. The reference screenshots in the PM issue were not useful to her — she works from acceptance criteria and product familiarity.
2. **The designer-to-engineer handover is unstructured.** Grace pings the assigned engineer when she's ready; the engineer then guides her through git/CLI operations she doesn't understand. There is no established handover process.
3. **Prototype-to-codebase sync is Grace's biggest pain point.** After designing in the prototype repo, the implementation in the actual code repo often looks visually different. Her current workaround is to take screenshots of both, place them side by side in Figma, and manually identify discrepancies — but she finds it too cumbersome to fix them all herself.
4. **Grace's design validation is user-driven, not specification-driven.** For major changes (new flows, or any divergence from the reference PG app), she user-tests with a minimum of 2-3 people, at least one of whom must be user-adjacent. Without user testing, she uses her own judgment and peer feedback. This is not a step an engineer can replicate without access to the same network.
5. **Grace explicitly prefers to hand the prototype to an engineer rather than implement it herself.** She is confident in the prototype repo but unconfident about anything that affects the production codebase. The gap she describes is not design judgment — it is the prototype-to-production transfer.

## Steps identified in designer process

| Step | Description | Engineer can do alone? | Notes |
|------|-------------|------------------------|-------|
| 1 | Read acceptance criteria to understand user needs | Yes | Engineers already do this |
| 2 | Review existing prototype repo to understand current state | Yes | Prototype repo is accessible to engineers |
| 3 | Prompt Claude Code in prototype repo to implement the design (referencing component library and design standards) | Partial | Engineer can prompt Claude, but lacks designer's instinct for which design system patterns to apply and how to phrase aesthetic constraints |
| 4 | Review generated design for micro-interaction correctness (alignment, spacing, column flush, etc.) | Partial | An engineer can check obvious issues but may not notice or prioritise the same details Grace does |
| 5 | Refine user flows and journeys with ChatGPT to verify they make sense for the user | Partial | Engineer can reason about flows but lacks user empathy and domain knowledge Grace brings |
| 6 | User test major changes (min. 2-3 people, at least one user-adjacent) | No | Engineers do not have the same access to users or user-adjacent colleagues; this is owned by design/PM |
| 7 | If no user testing: solicit feedback from peer designers (e.g. Lay Hui for copy) and user-adjacent colleagues (e.g. Hui Min) | No | Requires established relationships and design-community norms engineers don't have |
| 8 | Hand off: ping assigned engineer, follow their guidance to create a feature branch from prototype | N/A | This step becomes the engineer's own work in a designer-free flow |

## Candidate skills or harnesses

- **Visual diff harness** — given two running apps (prototype repo and code repo), take screenshots of corresponding views, diff them, and report specific visual discrepancies (font size, spacing, alignment). Directly addresses Grace's biggest pain point and could replace her manual Figma side-by-side comparison.
- **Prototype-to-codebase sync skill** — pull design changes from the prototype repo into the code repo, adapting to the actual codebase's component structure. Grace currently does this ad hoc by prompting Claude with the repo URLs; a structured skill could make this reliable and repeatable.
- **Design standards check skill** — given an implemented UI, check it against the project's component library and design standards and surface violations (micro-interaction issues, alignment, token misuse). Covers the review step (Step 4) that Grace currently does manually.
- **User flow validation skill** — given acceptance criteria, verify that the implemented flows match expected user journeys, surfacing gaps before user testing. Could reduce the minimum viable user-testing load for engineers working without a designer.

## Open questions

- What does Grace's Claude prompt look like when she references the prototype repo and design standards? Is it a consistent prompt she reuses, or ad hoc each time? (A reusable prompt structure could be the seed of the prototype sync skill.)
- How does Grace decide when a design change diverges enough from the reference PG app to require user testing? Is there a rule, or is it judgment?
- Do Reza and Wondo follow the same prototype-repo-first process, or do they use Figma? The handover problem and the prototype sync problem may look different depending on the designer.
- What does "follow established component library/design standard" mean in practice — is there a document Grace points Claude to, or is it in the prototype repo itself?
- Is the prototype repo always in sync with production, or does it sometimes lag? If it lags, the "review existing prototype" step (Step 2) may give an engineer a misleading starting point.
